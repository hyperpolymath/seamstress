== Claim 1: Architectural Seams as First-Class Artifacts

[quote, README]
____
Seamstress: Architectural seams as first-class artifactsâ€”documented, checkable, evolvable
boundaries between components.
____

*How it works:* A seam is recorded as a `*.seam.json` file validated against
`seams/schema/seam-record.schema.json`. The schema requires 22 top-level fields including `id`,
`title`, `status`, `owners`, `side_a`, `side_b`, `boundary_type`, `data_flows`,
`contract_artifacts`, `compat_policy`, `semantic_invariants`, `security_invariants`,
`privacy_invariants`, `test_vectors`, `checks`, `slo`, `failure_behavior`, `observability`,
`change_process`, and `rollout_backout`. Each record is a machine-checkable specification of a
boundary between two components. The canonical seam truth engine is `tools/seamctl` (Rust), which
validates records and generates reports. `services/seamstressd` (Elixir, optional) orchestrates
automated checks and future agentic workflows without being allowed to redefine seam semantics â€”
that responsibility stays exclusively in `seamctl`.

*Honest caveat:* `seamctl` is at an early stage; the HANDOVER.adoc describes the intended repo
layout in full but several tool subdirectories are scaffolding rather than implemented code. The
authoritative seam validation logic in Rust is the critical path and is not yet complete.

== Claim 2: Five Seam Type Taxonomy

[quote, SEAMS.adoc]
____
Seam types: Contract, Translation, Containment, Governance, Interaction
____

*How it works:* `SEAMS.adoc` defines five orthogonal seam types: Contract (schemas, IDLs, API
surfaces), Translation (representation mapping, adapters), Containment (isolation, retries,
timeouts, backpressure), Governance (ownership, approvals, rollout/backout), and Interaction
(human-machine boundaries: consent, reversibility, audit). A seam record's `boundary_type` field
must be one of these five, which constrains what `compat_policy`, `semantic_invariants`, and
`checks` fields need to contain. The taxonomy allows `seamctl` to apply type-specific validation
rules, so a Containment seam gets checked for backpressure SLOs while a Governance seam gets
checked for owner-approval records.

*Honest caveat:* The type-specific validation rules in `seamctl` are not yet fully implemented.
The JSON schema enforces structural validity; semantic validation per seam type requires additional
Rust logic that is specified in HANDOVER.adoc but not yet coded.
